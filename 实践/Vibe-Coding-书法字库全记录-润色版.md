# Vibe Coding 全记录：一个 PM 用 AI 交付书法字库系统

> **项目**：书法字帖 OCR + 字库管理 + 集字排版系统
> **工具**：OpenCode (DeepSeek V4 Flash Free) / MiMo Code
> **时间**：2026 年 5 月 – 至今
> **作者角色**：PM + DS（具备项目架构与深度学习基础理解）
> **仓库**：[github.com/LL0mc/calligraphy_chopping_tool](https://github.com/LL0mc/calligraphy_chopping_tool)
> **本文目的**：记录一次 Vibe Coding 项目的过程——选型、踩坑、决策与思考。

---

## 一、动机

练习硬笔行楷时，我发现了几个核心需求：

1. **单字学习**：字帖中同一汉字有多种写法，但 PDF 难以快速检索和对照，无法系统学习字形变化规律。
2. **作品排版**：手动排版费时费力，无法预览效果，需要自动化工具。
3. **字库积累**：未来可能练习不同字帖，需要建立可扩展的字库，支持跨书法家的字形对比。

市面上的工具要么缺少所需字帖，要么不支持自动切割和个性化定制。

**结论**：自己构建字库系统，在此基础上扩展功能。

## 二、开发历程

整体思路清晰：OCR 从 PDF 提取单字 → 切割入库 → Obsidian 管理 → 搜索、浏览、拼排版。

**瓶颈**：前后端交互技术栈缺失——学习成本高。

初期尝试 OpenCode + MiniMax、VSCode Agent 均未成功。后获 DeepSeek Flash 免费接口，效果显著提升，项目正式启动。

### Phase 1：方案探索

与 AI 讨论顶层架构：流程环节、可选方案、优劣对比。明确整体后，**优先启动单字识别模块**——与后续解耦，决定项目可行性。

### Phase 2：切割模块——最痛苦的阶段

**核心问题**：单纯 OCR 无法满足需求。

- 手写行楷识别率低，缺乏针对性训练
- 单字定位不精准，切割效果影响排版质量
- 字帖中的小字注释无法自动区分

**解决方案**：引入书法列提取、连通域精炼等辅助模块。

> **关键洞察**：AI 无法独立完成精确切割。需要人工校对环节，先调整满意再入库。

**反思**：若一开始就构建人工调整模块，快速生成 Ground Truth，是否会更高效？对能力边界的快速认知，能让开发过程更快收敛。

### Phase 3：交互模块——先做个垃圾再说

交互模块的难点在于：需求细节、状态管理、前后端逻辑——这些都需要通过实际 demo 迭代细化。

**策略**：快速生成原型 → 场景测试 → 反馈优化。

> **核心发现**：明确的需求 + 明确的测试用例 = 高效开发。但现实中难以一蹴而就，尤其新项目需要持续循环反馈。

**关于前端页面的补充**：我具备前端经验，能判断交互方式（表单、选项等）。缺乏经验时，如何引导模型提出恰当问题帮助用户思考，同样重要。

**描述能力的瓶颈**：与 Agent 的交互高度依赖描述清晰度。Lovable 等工具在静态调整上表现不错，但动态 UX 问题仍需人工描述。

### Phase 4：前端重设计——开源后的体面

开源后页面需要专业感。通过前端 Skill 让 Agent 在新分支调整，基本格式统一（圆角、间距等）。

**局限**：模型缺乏视觉审美，点击框颜色与背景混淆、交互 bug 等细节问题仍需人工修复。

### Phase 5：自动化迭代——让 AI 自己优化自己

基于已校对的 Ground Truth，尝试让 Agent 自动优化脚本。

**流程**：定义 Ground Truth → 建立指标 → Agent 迭代优化 → 多轮无提升则停止 → 记录过程。

**初期尝试**：小参数调整效果不佳。扩大调整维度后仍不理想。

**关键判断**：到这里我意识到——瓶颈不是参数微调，是 OCR 模型本身的识别能力。参数再怎么改，改不了 RapidOCR 对行书不同写法的盲区。所以方向从"修管线"转向"改 OCR"，尝试不同模型、考虑微调。这个判断不是 AI 做的，是人在全盘审视迭代结果后做的。

> **模型特性差异**：先后用过三个——Qwen 比较敷衍，DeepSeek 更喜欢自主探索，MiMo Code 在持续质疑下表现更佳。同样的 Agent 架构，不同 AI 效果差异明显。如何降低"人"这个变量的干扰，仍需思考。

## 三、踩坑记录

以下是实际协作中反复遇到的真实问题。

### 1. AI 在长任务中易迷失，存档至关重要

上下文压缩后，AI 常忘记决策、重复错误。压缩前等待决定，压缩后自行执行。

**解决方案**：
- 代码通过 Git 版本控制，效果达标即 commit
- 关键决策与错误原因记录到文档
- AI 自动版本标号，追踪各版本思路
- 维护 README/pipeline 实时反映方案状态

**思考**：长程任务的上下文压缩是脆弱时刻，如何保持状态无缝衔接？如何避免重复错误——需通过上下文调整强化约束。

### 2. 反馈描述需具体清晰

AI 会猜你的意图然后多做一些，但常常猜错或猜多。具体反馈 = 切实可行的下一步。

**问题**：当前交互流程仍显繁琐。虽然最终传递给 LLM 的是文本，但人侧如何用更高效的方式表达，是产品需要思考的方向。

### 3. 一次只反馈一个问题

同时反馈多个问题，AI 容易改错——丢西瓜捡芝麻。单点聚焦效果稳定。

**原因**：缺乏测试用例约束 + 模型注意力限制。

### 4. AI 不会主动拓宽方案边界

OCR 效果差时，AI 倾向于修补而非引入新方法（如 OpenCV 连通域分析）。需人工明确指示"试试这个方法"。

**改进尝试**：在项目设定中写入文档，但效果有限。Agent 存在思维惯性，后续考虑调整全局设定。

### 5. 单元测试缺失的代价

初期无单元测试，单人项目勉强可接受，但每次修改可能引入回归 bug。

**改进**：后续项目在深入调整前，先让 Agent 辅助建立基本测试项。

### 6. 完成 80% 就往前走

死磕完美易进入循环。有些问题下一阶段自然解决，或根本不是问题。

### 7. 行动失败的透明性问题

AI 执行失败后常不告知具体失败点和下一步。需让其自主分析并探索可行方案——这是 harness 处理特殊场景的问题。

### 8. 不同 AI 的特性差异

DeepSeek 倾向自主探索，Qwen 较敷衍。同样的 Agent 架构，不同 AI 效果不同。如何降低人的变量干扰，仍需思考。

## 四、Vibe Coding 方法论

基于本次项目实践，总结以下协作方法论：

### 项目启动

1. **先建仓库，再讨论设边界**：提前建仓库方便 commit 管理。和 Agent 讨论大致 scope，避免做到一半方向不对。细节后续迭代。

2. **让 Agent 先记录，形成项目概览**：讨论完后，让 Agent 把共识写成文档。这既是备忘，也让它建立对整个项目的理解。

### 过程管理

4. **分步骤，分成几个单元**：不要一下子都做成。拆成可独立交付的小模块，每个模块做完、测试通过、commit，再做下一个。

5. **不要让 Agent 猜，要具体描述**：看到方向不对要及时指出。模糊描述 = AI 猜错 = 返工。

6. **一次只改一个变量**：同时改多个东西，出问题不知道是哪里的锅。

7. **重要决策写下来**：AI 在长上下文中会遗忘，关键决策记录到文档，下次对话还能读到。

### 质量控制

8. **早期建立目标和测试用例**：否则你会不停手动测试、反馈，变成 Agent 的工具。有测试用例，Agent 可以自验，效率倍增。

9. **明确输出格式要求**：代码风格、文件结构、命名规范等，提前说清楚，否则每次都要纠正。

### 与 Agent 协作

10. **长程任务前确保 Agent 理解意图**：先让它用自己的话复述要做什么，确认理解一致后再开始。可以先做一轮小范围试试。

11. **大胆质疑，大胆 push**：不要被 Agent 骗到——它们倾向于放弃、倾向于说"做不到"。不合理的地方要追问，很多时候 push 一下就有解。

12. **鼓励 Agent 跳出思维框架**：AI 习惯于在当前方案下不断修正细节，缺少对全局的审视和灵活切换方向的能力。当迭代多轮没有起色时，需要人介入，让它退一步想想——"是不是方向错了？有没有其他方案？"

13. **常犯的错误让它记下来**：错误是 Agent 的惯性。把常见错误写进上下文提示，能有效减少重复犯错。

14. **做完一系列任务后，让 Agent 整理文档，compact 上下文**：模型的关注能力有限，主动管理上下文，速度变快，效果更好。

### 心态调整

15. **版本控制要及时**：效果差不多就 commit，方便回退。AI 可能一个改动就把之前好的东西改坏。

16. **完成 80% 就往前走**：死磕完美易进入循环。有些问题下一阶段自然解决，或根本不是问题。

17. **Agent 前后说法不一致是常态**：不要因此生气，让它自己重新理清就好。毕竟是免费的，付费的另说。

## 五、关键决策

1. **保留人工校对环节**——纯 AI 管线卡在长尾问题上，PM 经验判断需人工介入。
2. **优先单字识别模块**——与后续解耦，决定项目可行性。
3. **未立即微调模型**——先参数/逻辑修补，发现全部退步后重新审视瓶颈——不是微调，而是 OCR 本身。

## 六、成果

```
PDF 字帖 → 渲染 → 内容裁剪 → OCR 字级检测
    → 连通域精炼（修复飞白断笔）→ 去重
    → 人工校对界面（拖拽修正/增删字符）
    → 切片存储 → 字库浏览 + 集字排版 + Obsidian 字库
```

**三个 Flask Web 应用 + 一个 Pipeline**，覆盖全流程。

- 第一版一周内完成：校对、浏览、排版三个应用
- 解决真实 OCR 问题：飞白断笔合并、远距离捺笔保留、列尾墨点排除、后字窃取笔画等
- Obsidian 字库集成：每字一条笔记 + 变体嵌入
- 持续改进中：横排支持、不同模型、微调

## 七、我的体会

大部分时间并非写代码，而是：
1. **人形测试反馈机**——每页输出问题 → 归纳 → 反馈 AI
2. **架构决策者**——方案选择、人工介入时机、优先级排序
3. **评价标准定义者**——好不好用靠指标说话

> **核心认知**：AI 是执行者，不是大脑。项目方向、方案选择、质量控制——必须由人负责。AI 擅长把确定的任务做快、做好。

**更深体会**：当前与 AI 的反馈是断点的——我处理它给的事情，它处理我的反馈。若 AI 能提前拆解可测试单元主动推送，效率会更高。这是未来 AI 协作工具的改进方向。

Agent 很像工厂自动化——人的行为被 Agent 重新限定，构成新的分工模式。期待这一天的到来。
