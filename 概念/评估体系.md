
## 评估指标

1. 任务成功类：

| 指标                              | 定义                   | 如何测量                         | 面试常见追问                                 |
| ------------------------------- | -------------------- | ---------------------------- | -------------------------------------- |
| **任务完成率（Task Completion Rate）** | Agent成功完成用户明确目标的对话比例 | 离线人工标注 + 在线用户确认（如“问题解决了吗”弹窗） | “成功”怎么定义？用户中途离开算失败吗？<br>（完全成功、部分成功、失败） |
| **人工干预率（Human Handoff Rate）**   | 需要转人工或用户主动结束对话的比例    | 线上日志：转人工事件数 / 总会话数           | 太低可能意味着Agent冒险乱答，太高则成本高                |
| **用户满意度（CSAT / Follow-up）**     | 用户评价或是否继续使用          | 问卷或“是否推荐”打分                  | 如何区分是Agent问题还是产品整体问题？                  |

2. 效率类   

| 指标                                  | 定义                  | 测量方式    | 关注点                                                    |
| ----------------------------------- | ------------------- | ------- | ------------------------------------------------------ |
| **平均交互轮次（Avg. Turns）**              | 完成任务所需的对话次数         | 线上统计    | 轮次太少可能跳过必要信息；太多则啰嗦                                     |
| **工具调用有效率（Tool Call Success Rate）** | 调用的工具中成功返回预期结果的比例   | 离线/线上日志 | 模型参数写错导致调用失败是常见问题                                      |
| **首次行动延迟**                          | 用户说完后到Agent第一次动作的时间 | 线上统计    | 太长会让人以为卡死了<br>注意这里不是首次Token延迟Time to First Token, TTFT |
关于首次行动延迟和首次Token延迟，可以看是以什么为主的Agent：
如果是对话为主，则是TTFT，如果是执行为主，则是首次行动
根据具体的类型来划分重要节点：比如Coding，首次行动可以是开始写文本计划，或者是开始写代码。可以根据锚定的一些具体中间过程来确定。
此外，在交互上，要记得给出进度提示，避免因为延迟产生焦虑

Time to Last Token (TTLT) 从用户发出消息到Agent完全输出本轮所有内容的时间。（按回合计算）太长会引起用户中断

3. 安全/体验类

| 指标                              | 定义                   | 备注                        |
| ------------------------------- | -------------------- | ------------------------- |
| **拒绝率（Refusal Rate）**           | Agent合理拒绝有害/越权请求的比例  | 太低表示护栏失效；太高可能过度防御         |
| **[[幻觉]]率（Hallucination Rate）** | Agent声称做了某事但实际没做/做不到 | 对Agent尤其危险（用户信任崩塌）        |
| **可解释性得分**                      | Agent是否清晰说明了行动理由     | 通常通过人工抽样打分 / LLM（也是需要校准的） |


4. 成本类

| 指标               | 定义              |
| ---------------- | --------------- |
| **每任务平均Token数**  | 输入+输出token      |
| **每任务平均API调用次数** | 包括模型调用和工具调用     |
| **每任务平均成本（美元）**  | 综合模型、外部API、人工兜底 |

## 评估集的构建

需要包括各种case：
1. 黄金案例：最常见、最重要的场景
2. 边界案例：模糊指令、缺失信息、多意图混合
3. 对抗案例：恶意、不合理的
4. 长尾案例：失败的、低评分的、转人工的

需要标注：
1. 期望的工具调用序列
2. 期望的最终答案/动作
3. 关键中间状态

对于开放性强的Agent，主要看结果导向 + 关键约束

## 如何自动化评估

1. 模拟环境执行：自动化过程中，会用另一个LLM来扮演用户（用户模拟器/评估模型），通过给他必要的一些设定（角色扮演，环境，目标判断），同时生成多种变体来进行交互测试。
2. 规则检查
3. 模型评分：需要精确的评分标准，需要和人工校准
4. 抽几条人工复核校准



## 如何定位问题

1. 下钻分析：按照错误类型、用户意图、时间段/模型版本
2. 抽取bad case，来进行分析和标注
3. 归因分析：
	1. 模型能力问题 - 工具参数问题、推理链条断裂
	2. 工具设计缺陷 - API设计的不好
	3. 产品规则模糊 - 决策链条
	4. 环境问题 - 服务不稳定
	5. 用户问题 - 用户输入乱码、反复修改
4. 指定改进策略
