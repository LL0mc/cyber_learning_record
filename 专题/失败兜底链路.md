## 失败的类型

|   类型    | 原因                               |
| :-----: | -------------------------------- |
| 工具调用失败  | 外部系统不可靠，API、网络问题、格式错误、权限问题       |
|  推理失败   | 规划错误，模型能力不足 / 提示词问题 比如选错工具、参数选错  |
|  输入失败   | 用户给出的信息矛盾，输入一些乱码等                |
| 安全/约束触发 | 护栏拦截危险行为                         |
|  资源失败   | Token超限、内容、成本等等                  |
| 状态/记忆丢失 | 上下文丢失、记忆错乱（上下文管理逻辑导致的，或者用户行为导致的） |


## 处理的思路

1. 重试策略：什么情况下触发，重试几次，间隔多少，重试后的下一步
2. 降级：功能退化，尝试别的路径
3. 超时和熔断
4. 用户通知和解释

## 处理的手段。

| 手段                         | 适用场景                | 示例                                    |
| -------------------------- | ------------------- | ------------------------------------- |
| **重试（Retry）**              | 临时性故障（网络抖动、服务短暂不可用） | 调用日历API超时，等1秒后重试，最多3次<br>（[[指数退避]]）   |
| **降级（Fallback）**           | 主功能不可用，用替代方案完成部分目标  | 查不到实时航班，改用历史时刻表查询                     |
| **转人工（Handoff）**           | 不可恢复或高风险任务          | Agent反复确认用户意图失败，说“我帮你转到人工客服”          |
| **拒绝并解释（Reject + Reason）** | 安全或权限失败             | “抱歉，我不能帮你删除公司邮件，这需要手动操作”              |
| **补问（Re-prompt）**          | 信息不足或模型输出异常         | Agent调用工具返回空数组，则追问“没有找到结果，你要不要换个关键词？” |
| **熔断（Circuit Break）**      | 成本或资源失控             | 单次对话调用工具超过10次，强制终止并提示用户简化需求           |


## 实现方法

这些逻辑大部分是写死的，但越来越多公司把部分兜底交给Agent自主决策

决策依据：
1. 涉及到系统底线（安全、成本、稳定性），一旦失控后果比较严重的，要写死来兜底。或者确定性的错误，也可以写死
2. 动态决策的部分，适合用模型，比如，降级、补问、转人工。但也要设计兜底的兜底。

模型如何自己决策：
1. Prompt：低风险，逻辑性的
2. 工具：结构化的输出，而且可以预设一些工具，不允许模型自己疯狂重试
